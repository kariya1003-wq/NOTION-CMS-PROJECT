# 개발 로드맵: 개인 개발 블로그 (Notion CMS 기반)

## 📋 개요

이 로드맵은 PRD의 요구사항을 바탕으로 합리적인 개발 순서를 제시합니다.
**핵심 원칙**: 기초부터 쌓아 올리되, 최대한 빨리 동작하는 최소 기능(MVP)을 완성한 후 반복적으로 개선합니다.

---

## Phase 1️⃣: 프로젝트 초기 설정 (골격 구축)

**기간**: ~3일 | **우선순위**: 🔴 필수

### 목표
- 개발 환경 완성
- Notion 연동 준비
- 프로젝트 기본 구조 확립

### 작업 항목

#### 1-1. 환경 및 패키지 설정
- **작업**: Next.js 프로젝트 초기 설정, 필수 패키지 설치
  ```bash
  npm install @notionhq/client
  npm install -D @types/node
  ```
- **이유**: Notion API 클라이언트 없이 블로그 시작 불가능
- **완료 기준**:
  - ✅ 패키지 설치 완료
  - ✅ `npm run dev` 실행 시 에러 없음

#### 1-2. 환경 변수 설정
- **작업**: `.env.local` 작성
  ```
  NOTION_API_KEY=your_integration_key
  NOTION_DATABASE_ID=your_database_id
  ```
- **이유**: 보안상 API 키를 코드에 하드코딩하면 안 됨
- **완료 기준**:
  - ✅ `.env.local` 파일 생성
  - ✅ Notion Integration 생성 및 키 확보

#### 1-3. Notion 데이터베이스 구성
- **작업**: Notion에 블로그 데이터베이스 생성
  - Title (텍스트)
  - Category (단일 선택)
  - Tags (다중 선택)
  - Published (날짜)
  - Status (단일 선택: 초안/발행됨)
  - Content (페이지 본문)
- **이유**: CMS의 데이터 구조가 정해져야 쿼리 로직을 작성할 수 있음
- **완료 기준**:
  - ✅ 데이터베이스 필드 생성
  - ✅ 샘플 글 3~5개 작성 및 "발행됨" 상태로 설정
  - ✅ Integration이 데이터베이스에 접근 가능 확인

#### 1-4. 프로젝트 구조 생성
- **작업**: 기본 디렉토리 구조 생성
  ```
  app/
  ├── layout.tsx          # 루트 레이아웃
  ├── page.tsx            # 홈
  ├── posts/
  │   └── [slug]/
  │       └── page.tsx
  └── categories/
      └── [category]/
          └── page.tsx
  lib/
  ├── notion.ts           # Notion 클라이언트
  ├── posts.ts            # 비즈니스 로직
  └── utils.ts
  ```
- **이유**: 일관된 구조로 나중에 수정하는 시간 절감
- **완료 기준**:
  - ✅ 디렉토리 구조 생성
  - ✅ 모든 페이지에 `layout.tsx` 적용 확인

## Phase 2️⃣: 공통 모듈/컴포넌트 개발

**기간**: ~4일 | **우선순위**: 🔴 필수

### 목표
- Notion API 추상화 계층 구축
- 재사용 가능한 컴포넌트 준비
- 데이터 흐름 설계

### 작업 항목

#### 2-1. Notion API 래퍼 작성
- **작업**: `lib/notion.ts` 구현
  ```typescript
  // Notion 클라이언트 초기화 및 설정
  ```
- **이유**:
  - Notion API의 복잡한 응답 구조를 단순화
  - 향후 데이터베이스 쿼리 변경 시 수정 포인트 최소화
- **완료 기준**:
  - ✅ 클라이언트 초기화 코드 작성
  - ✅ API 호출 테스트 가능 (콘솔에서 확인)

#### 2-2. 글 조회 함수 작성
- **작업**: `lib/posts.ts` 구현
  ```typescript
  // getPosts(category?): 발행된 글 목록
  // getPostBySlug(slug): 특정 글 조회
  // getPostContent(pageId): 글 본문 블록 조회
  // getCategories(): 카테고리 목록
  ```
- **이유**:
  - 복잡한 Notion 쿼리 로직을 페이지 컴포넌트와 분리
  - 테스트와 유지보수 용이
- **완료 기준**:
  - ✅ `getPosts()` 함수로 발행된 글 목록 조회 가능
  - ✅ 필터링(카테고리) 동작 확인
  - ✅ 정렬(Published 내림차순) 확인

#### 2-3. UI 컴포넌트 추가
- **작업**: shadcn/ui 컴포넌트 설치 및 커스텀
  ```bash
  npx shadcn@latest add card
  npx shadcn@latest add button
  npx shadcn@latest add input
  npx shadcn@latest add badge
  ```
- **이유**: 스타일링 작업 시간 단축, 일관된 디자인 시스템
- **완료 기준**:
  - ✅ 필요한 컴포넌트 설치 완료
  - ✅ 각 컴포넌트 스토리 확인

#### 2-4. 레이아웃 및 Header 컴포넌트
- **작업**: `components/layout/Header.tsx` 구현
  - 로고, 블로그 제목, 네비게이션
- **이유**: 모든 페이지가 공통 헤더를 사용할 것이므로 먼저 구현
- **완료 기준**:
  - ✅ 반응형 헤더 렌더링 확인
  - ✅ 네비게이션 링크 동작 확인

## Phase 3️⃣: 핵심 기능 개발

**기간**: ~5일 | **우선순위**: 🔴 필수

### 목표
- MVP 기능 완성
- 사용자가 실제로 블로그를 사용 가능한 상태

### 작업 항목

#### 3-1. 홈 페이지 구현 (`/`)
- **작업**: 글 목록 + 카테고리 필터 UI
  - `app/page.tsx` 작성
  - 글 카드 컴포넌트
  - 카테고리 탭 필터
- **이유**: 사용자의 첫 진입점, 가장 중요한 페이지
- **완료 기준**:
  - ✅ Notion에서 글 목록 조회하여 렌더링
  - ✅ 카테고리 필터 동작 (URL 파라미터: `?category=React`)
  - ✅ 최신 순서대로 정렬 확인
  - ✅ 글 카드에 제목, 카테고리, 태그, 발행일 표시

#### 3-2. 글 상세 페이지 구현 (`/posts/[slug]`)
- **작업**:
  - `app/posts/[slug]/page.tsx` 작성
  - Notion 페이지 블록 파싱 및 렌더링
  - 메타데이터(제목, 카테고리, 태그, 발행일) 표시
- **이유**: 글을 읽기 위한 필수 기능
- **완료 기준**:
  - ✅ 글 상세 페이지 동적 라우팅 동작
  - ✅ Notion 본문(제목, 단락, 코드 블록 등) 렌더링
  - ✅ 메타데이터 정확히 표시
  - ✅ 이전/다음 글 네비게이션 표시

#### 3-3. 카테고리 페이지 구현 (`/categories/[category]`)
- **작업**:
  - `app/categories/[category]/page.tsx` 작성
  - 홈 페이지와 동일한 카드 레이아웃
- **이유**: 사용자가 특정 주제의 글들을 한눈에 볼 수 있도록
- **완료 기준**:
  - ✅ 카테고리별 글 목록 조회 및 렌더링
  - ✅ 동적 라우팅 동작

#### 3-4. 반응형 디자인 검증
- **작업**:
  - 모바일(320px), 태블릿(768px), 데스크톱(1024px) 테스트
  - Tailwind CSS 반응형 유틸리티 적용
- **이유**: 다양한 화면 크기에서 UX 저하 방지
- **완료 기준**:
  - ✅ 모든 페이지가 주요 해상도에서 정상 렌더링
  - ✅ 모바일에서 글 읽기 편함
  - ✅ 레이아웃 시프트(CLS) 없음

## Phase 5️⃣: 최적화 및 배포

**기간**: ~3일 | **우선순위**: 🔴 필수

### 목표
- 프로덕션 환경에 배포
- 성능 및 SEO 최적화

### 작업 항목

#### 5-1. ISR(Incremental Static Regeneration) 설정
- **작업**:
  - 홈 페이지, 글 상세 페이지에 `revalidate` 설정 (60초)
  - Notion 데이터 업데이트 시 캐시 갱신 전략
- **이유**:
  - SSG의 빠른 로딩 속도 + 최신 콘텐츠 반영
  - Vercel 자동 배포로 수동 개입 최소화
- **완료 기준**:
  - ✅ ISR 설정 완료 (revalidate: 60초)
  - ✅ 로컬에서 ISR 동작 확인
  - ✅ Notion 업데이트 후 60초 내 블로그 반영

#### 5-2. SEO 최적화
- **작업**:
  - `next/head` 또는 `Metadata` API로 메타태그 설정
  - 모든 페이지에 `title`, `description`, `og:*` 태그 추가
  - `sitemap.xml`, `robots.txt` 생성 (선택)
- **이유**: 검색 엔진 최적화, 소셜 공유 개선
- **완료 기준**:
  - ✅ 모든 페이지의 메타데이터 설정
  - ✅ og:image 설정 (소셜 공유 썸네일)
  - ✅ Google Search Console 등록 및 색인 확인

#### 5-3. 성능 최적화
- **작업**:
  - 이미지 최적화 (`next/image` 사용)
  - CSS-in-JS 제거 (이미 Tailwind 사용)
  - 번들 크기 분석 및 최적화
  - LCP, FID, CLS 개선 (2.5초 이하 목표)
- **이유**: 사용자 경험 향상, 검색 엔진 순위 개선
- **완료 기준**:
  - ✅ Lighthouse 점수 90점 이상
  - ✅ LCP < 2.5초
  - ✅ 불필요한 번들 제거

#### 5-4. 접근성(A11y) 검증
- **작업**:
  - 스크린 리더 테스트
  - 색상 대비 확인 (WCAG AA)
  - 키보드 네비게이션 확인
  - `alt` 텍스트, ARIA 레이블 추가
- **이유**: 모든 사용자가 콘텐츠에 접근 가능하도록
- **완료 기준**:
  - ✅ axe DevTools 또는 WAVE로 접근성 이슈 0개
  - ✅ WCAG 2.1 AA 수준 달성

#### 5-5. Vercel 배포
- **작업**:
  - Vercel 계정 생성 및 프로젝트 연동
  - GitHub 연결 (자동 배포 설정)
  - 환경 변수 설정 (NOTION_API_KEY, NOTION_DATABASE_ID)
  - 커스텀 도메인 설정 (선택)
- **이유**: 무료 호스팅, Next.js 최적화, 글로벌 CDN
- **완료 기준**:
  - ✅ Vercel에 배포 완료
  - ✅ 프로덕션 URL 접속 가능
  - ✅ 글 목록, 상세 페이지 정상 렌더링
  - ✅ ISR 동작 확인 (Notion 업데이트 반영)

#### 5-6. 모니터링 및 에러 추적
- **작업**:
  - (선택) Sentry 또는 유사 서비스 연동
  - 프로덕션 에러 로깅 설정
- **이유**: 배포 후 문제 조기 발견
- **완료 기준**:
  - ✅ 에러 추적 서비스 활성화
